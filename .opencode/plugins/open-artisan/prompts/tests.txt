## Workflow Phase: TESTS

You are in the TESTS phase. Your goal is to write a comprehensive, failing test suite.

**Critical rule:** Tests must fail — they specify behavior that does not yet exist.
If any test passes at the end of this phase, that means implementation has leaked in.

### What the test suite must cover

1. **Coverage** — at least one test per interface method, one test per data model constraint
2. **Happy path** — the main success case for each operation
3. **Edge cases** — empty input, maximum values, null/undefined, boundary conditions
4. **Error cases** — network failures, invalid data, auth failures, timeouts, concurrency issues
5. **Naming** — test descriptions must match the corresponding interface specification exactly
6. **Independence** — each test must be able to run in isolation

### Sub-states

**DRAFT** — Write only test files appropriate to your project's language and test framework.
  Accepted patterns: `foo.test.ts`, `foo_test.go`, `test_foo.py`, `FooSpec.java`, files in
  `test/`, `tests/`, `__tests__/`, `spec/`, `specs/` directories, etc.
  - Import from the interfaces you defined — do not import implementation
  - Organize tests by interface or feature
  - Every test must assert behavior that will fail against a stub or unimplemented code
  - When the test suite is complete, call `request_review`.

**REVIEW** — (Isolated self-review checks test coverage against the interfaces)

**USER_GATE** — Present the test suite for user approval.
  Summarize: total test count, coverage by interface, notable edge cases.

**REVISE** — Add/modify tests based on feedback. Never remove passing tests.

### Constraints
- Write ONLY test files. No implementation files, no interface files, no markdown.
- No implementation code. No code that makes tests pass — tests must fail until IMPLEMENTATION.
- No .env file access. Bash execution is blocked in this phase — you cannot run commands. Write your tests and call `request_review` when done. Test execution will be verified during the IMPLEMENTATION phase.
- Tests must import from the interfaces module, not from implementations.
- Do not mock the interfaces themselves — mock only external dependencies (network, filesystem).
- If a conventions document was approved in DISCOVERY, follow its test conventions
  (framework, file naming, assertion style, mock patterns) exactly.
