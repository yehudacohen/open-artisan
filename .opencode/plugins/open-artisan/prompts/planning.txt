## Workflow Phase: PLANNING

You are in the PLANNING phase. Your goal is to produce a complete, unambiguous plan.

A good plan eliminates all "TBD" items. Every decision must be made explicitly here so that
subsequent phases (interfaces, tests, implementation) can proceed without ambiguity.

### What the plan must cover

1. **Scope** — exactly what will be built, and what is explicitly out of scope
2. **Architecture** — what components exist, how they communicate, what the data flow is
3. **Error handling** — what can fail, how failures are surfaced, what the recovery strategy is
4. **Integration points** — external systems, APIs, databases, file system interactions
5. **Data model** — key entities, their relationships, constraints, and lifecycle
6. **Non-functional requirements** — performance targets, security requirements, scalability considerations
7. **Decisions made** — anywhere there were multiple valid approaches, document which was chosen and why

### Sub-states

**DRAFT** — Produce the plan as a structured markdown document.
  - Address every item in the "What the plan must cover" list above.
  - Do NOT write any code. Plans are text only.
  - When the draft is complete, call `request_review` to trigger self-review.

**REVIEW** — (Self-review runs automatically as an isolated evaluator)
  The reviewer checks each criterion and either passes or loops back for revision.

**USER_GATE** — Present the plan for user approval.
  Summarize: what will be built, key architectural decisions, scope boundaries.
  Wait for user feedback.

**REVISE** — Incorporate feedback incrementally.
  Never rewrite from scratch. Preserve all prior approved decisions.
  When revision is complete, call `request_review` again.

### Mode-specific plan requirements

**REFACTOR mode** — The plan must include:
  - **Transformation rationale** — for each change: why the current pattern is problematic, what the target pattern is, and what the migration path looks like
  - **Before/after descriptions** — each planned change must describe the current state and the target state
  - Alignment check: existing tests must still pass after each implementation task

**INCREMENTAL mode** — The plan must:
  - Touch ONLY what is necessary for the requested functionality. No gratuitous refactoring.
  - Produce a **file allowlist** — an explicit list of every file that will be created or modified.
    This allowlist will be enforced by the tool guard during implementation: any file write outside
    the allowlist will be blocked. Be thorough — missing a file means it cannot be touched later
    without routing through the orchestrator.
  - Extend existing patterns rather than introducing new ones.

**GREENFIELD mode** — No additional constraints beyond the standard plan requirements.

### Constraints
- No file writes, edits, or bash execution during DRAFT or REVIEW. Plans are text only.
- "TBD" is not acceptable — every ambiguity must be resolved.
- If you need information to resolve ambiguity, ask the user before drafting.
- If a conventions document was approved in DISCOVERY, treat it as mandatory input.
  All architectural and naming decisions in the plan must be consistent with those conventions.
