## Workflow Phase: IMPLEMENTATION PLAN

You are in the IMPL_PLAN phase. Your goal is to produce a DAG of implementation tasks.

The implementation plan breaks the approved interfaces into concrete, atomic tasks with
explicit dependencies. Each task should be small enough to implement in one focused session
and large enough to have a meaningful testable outcome.

### What the implementation plan must contain

For each task:
1. **ID** — unique identifier (e.g., "task-001")
2. **Description** — what this task implements, which interface methods/types it covers
3. **Dependencies** — IDs of tasks that must complete before this one starts
4. **Expected tests** — which test file paths should pass when this task is complete
5. **Category** — one of: `scaffold`, `human-gate`, `integration`, `standalone` (see below)
6. **Complexity** — small / medium / large estimate
7. **Merge point** — if parallel tasks converge here, identify it

### Task categories

Every task must be categorized as one of:

1. **scaffold** — Writes the contract-satisfying code structure (types, module files,
   function signatures, constructor wiring) without real integration logic. Stubs are
   acceptable ONLY in scaffold tasks. Scaffold tasks should make tests compile but
   may not make all assertions pass.

2. **human-gate** — Requires human action (provision infrastructure, configure credentials,
   approve third-party service signups, set up external accounts). The agent cannot complete
   these — they are automatically held for user resolution at USER_GATE. These tasks have
   NO code — just a clear description of what the human must do and how to verify it.

3. **integration** — Writes the real implementation logic that connects to real services,
   databases, APIs. Integration tasks MUST depend on their corresponding scaffold task
   AND any relevant human-gate tasks. Integration tasks are where stubs get replaced
   with real code. Stubs are NOT acceptable in integration tasks.

4. **standalone** — Self-contained tasks that don't require external services (pure
   business logic, data transformations, utility functions). Stubs are NOT acceptable.
   This is the default category if not specified.

### Planning rules for categories

- If a task requires external credentials or infrastructure, it MUST be split into:
  (a) scaffold task → (b) human-gate task → (c) integration task
- A single task CANNOT be both scaffold and integration
- Human-gate tasks must list exactly what the human needs to do and how to verify it
- Integration tasks must explicitly list which scaffold stubs they replace
- The critical path analysis must account for human-gate delays
- Tasks that are pure business logic with no external dependencies should be `standalone`

### Deployment & infrastructure tasks

If the approved plan includes a "Deployment & infrastructure" section, the DAG MUST include
corresponding tasks. Common deployment concerns that require explicit tasks:

- **Infrastructure provisioning** (databases, queues, storage buckets, cloud services) → `human-gate`
- **Credential/secret setup** (API keys, connection strings, certificates) → `human-gate`
- **Environment configuration** (env vars, config files, feature flags) → `scaffold` + `human-gate`
- **Deployment pipeline changes** (CI/CD config, Dockerfiles, IaC templates) → `standalone` or `integration`
- **DNS / routing / certificates** → `human-gate`
- **Verification / smoke tests** → `integration` (depends on all deployment human-gates)

A DAG that implements all code but omits deployment tasks is incomplete. The final task in
the critical path should verify the feature works in the target environment, not just locally.

### DAG validity requirements

- No cycles (task cannot depend on itself, directly or transitively)
- Every interface method is covered by at least one task
- Merge points explicitly identified where parallel branches converge
- Expected test outcomes specified per task (which tests become green)
- Every `integration` task depends on its corresponding `scaffold` task
- Every `integration` task depends on relevant `human-gate` tasks for credentials/infra
- No `standalone` task depends on a `human-gate` task

### Sub-states

**DRAFT** — Produce the DAG as a structured markdown document with a task table.
  No code. No file writes.
  When the draft is complete, call `request_review` with the full impl plan text in `artifact_content`.
  This writes it to `.openartisan/impl-plan.md` immediately — the user can read it there.

**REVIEW** — (Isolated self-review validates DAG structure and coverage)
  Call `mark_satisfied` with your per-criterion assessment.
  The isolated reviewer reads `.openartisan/impl-plan.md` directly — do NOT pass `artifact_content` here.

**USER_GATE** — Present the implementation plan for user approval.
  Summarize: total tasks, parallel groups, critical path, expected test outcomes per task.
  Highlight any human-gate tasks that require user action.

**REVISE** — Refine the plan incrementally. Never remove a task without replacing its coverage.

### Constraints
- **No file writes, edits, or bash execution in this phase.** The plan is text only.
  Pass the plan as `artifact_content` in `request_review` — do NOT use the Write/Edit tools.
- Keep the plan focused and concise. Each task description should be 1-3 sentences,
  not a full specification. The goal is a clear task graph, not exhaustive prose.
  If your plan is very large, break it into fewer, larger tasks rather than many micro-tasks.
- The plan must cover every interface — no implementation gaps.
- If a conventions document was approved in DISCOVERY, follow its naming and structural conventions when defining tasks.
